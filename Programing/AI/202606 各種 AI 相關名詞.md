## Token


LLM 其實在處理過程中是不斷在跑個數學函式，其實根本不認識這我們 input 的文字。

我們每次 input 文字給他，中間會有一個中間人，叫做 Tokenizer，他專門做 decode + encode，文字轉成數字或反之

例如我們說「今天你過的好嗎？」可能會變轉成 [5, 99, 13, 8]，具體過程會把我們的文字做「切分」例如會把它切成 「今天」「你」「過的」「好嗎？」，這些切分的碎塊就叫做「Token」，再來我們就會 map 到特定數字 [5, 99, 13, 8]，這些數字就變成 Token ID，本質上也是一樣的意思，只是換種某種表達方式，然後把 Token ID 交給 LLM 後，他會開始文字接龍，output 出特定的 Token ID 最後 decode 回文字，這邊就不用切分了，所以我們每次對話都會看到他是一個字一個字顯示

但注意剛剛的切分，直覺上很像一個詞，所以可能會以為 Token = 詞，但其實不然，我們可以透過 OpenAI 提供的 Tokenizer 網站來測試看看 https://platform.openai.com/tokenizer

## Context

而我們在對話中有時候你會發現他可以記住我們前幾句話，明明 LLM 就是一個數學函式，所以應該沒有記憶，可是他卻記住之前的對話內容，答案就是每次會保存對話記錄，這樣每次 LLM 看到你的問題，就知道你前面有敘述過的事情，這也引發出上下文 Context 的概念，在每次處理任務時所接收到的訊息總合

所以一個 Context 包含

- 對話紀錄：你好，我叫 XXX
- 用戶問題：我叫什麼？
- 當前輸出：你叫 XXX
- 工具列表：天氣工具、定位工具（一些外部工具）
- System Prompt：設定背景，例如你是一個好聊天的人

也可以理解 Context 是 LLM 臨時的記憶體，那這也延伸出 Context Window (上下文窗口）的概念，代表 Context 能容納最大的「Token 數量」，例如上限是一萬，他最多就只能處理一萬個 Token

但現在的 LLM 都很強，例如

- GPT-5.4 - 105萬
- Gemini 3.1 Pro - 100萬
- Claude Opus 4.6 - 100萬

假如現在有一個公司產品的手冊（上百萬的字），希望 LLM 能針對他做問題回答，假如一次丟給 LLM，這不是一個最佳策略，因為這手冊內容真的太長了，即使 Context Window 沒有被塞爆，但也不好讓他掌控資訊，這情境就適合 RAG (Retrieval-Augmented Generation - 檢索增強生成) 技術了，他可以從手冊中抽取用戶問題最匹配的幾個片段，然後只把這幾個片段抽給 LLM，讓 LLM 只根據這幾個片段來回答問題，這樣 LLM 就不是接收一大堆內容，而是幾段話，這樣就不受 Context Window 大小的限制了，成本也低很多

RAG 細節可參考 https://youtu.be/WWdlme1EAGI?si=4dnfRcQ_SwgftzbD

## RAG

RAG 的全名為 **Retrieval-Augmented Generation（檢索增強生成）**。這是一項結合了「資訊檢索系統」與「大型語言模型（LLM）文字生成能力」的核心 AI 技術。

想像 LLM 是一個絕頂聰明、讀過很多書的學生，但他腦海中的知識停留在畢業（模型訓練完成）的那一天，且有時候會因為記憶模糊而亂掰答案（AI 幻覺）。

RAG 技術就像是給這個學生一場「開卷考試（Open-book exam）」的權利。當你問他問題時，他不會只死背答案，而是先去「圖書館（外部資料庫）」精準查閱相關的書籍段落，然後綜合這些正確的參考資訊來回答你。

### 1. 資料準備與索引 (Indexing)

我們不會讓 LLM 直接閱讀幾十萬頁的 PDF 或整個維基百科，而是先在背景將龐大數據結構化：

- **切塊 (Chunking)：** 將長篇文件切分成一個個數百字的小段落。
- **向量化 (Embedding)：** 使用 Embedding 模型，將這些文字段落轉換成數學上的「向量」（一串高維度的數字矩陣）。
- **儲存：** 將這些向量存入專門處理此類數據的 **向量資料庫 (Vector Database)** 中

### 2. 檢索 (Retrieval)

當用戶向系統提出問題（例如：「公司最新的請假規定是什麼？」）時：

- 系統會先把用戶的「問題」也轉換成數學向量。
- 接著在向量資料庫中進行「相似度搜尋」，計算問題向量與資料庫中所有段落向量的數學距離。
- 系統會精準 **擷取 (Retrieve)** 出語意最接近、最相關的 Top-K 個段落（例如撈出員工手冊第 15 頁的具體規定）。

### 3. 生成 (Generation)

- 系統會將用戶的「原始問題」以及剛剛找出來的「相關參考資料」一起打包成一個提示詞（Prompt）。
- 送交給 LLM，背後的指令通常類似：「*請根據以下提供的參考資料，回答用戶的問題。如果參考資料中沒有答案，請說不知道。*」
- 最後，LLM 發揮它的強項，**生成 (Generate)** 出流暢、邏輯清晰且有憑有據的最終答案。

## Prompt

Prompt 是 LLM 接收具體問題或指令的東西，例如當你提問，「幫我想一首詩」這就是 prompt，但假如問題太模糊，就很容易產出不預期的結果，像上述的提問，可能會出現古詩、現代詩都有可能，所以一個好的 prompt 是具體明確的，所以以前 23-24 年的時候，有專門的職缺是 Prompt 魔法師，希望能下好的 prompt，而這專業的行為就稱為 Prompt Engineering，就是要把我們人說的意圖說清楚給 AI 好理解，但到現在 Model 都很強，也加本身上門檻較低，現在比較不流行了，不需要花太多功夫在 prompt 上了。

但除了每次我們需要和 AI 說具體任務外，有時候還需要幫她做人設或做事規則，告訴 LLM 他是誰，這引起兩種 Prompt

- User Prompt 具體任務
- System Prompt 人設和做事規則

假設你現在想要做一個「教學型的機器人」這時候就可以用到這兩種 prompt，因為要人設並且給他規則，可以用 System Prompt，內容：每次 user 輸入問題都不能直解告訴答案，要引導他，幫助他理解問題，培養解決問題的能力。

後續在使用時就是 User Prompt，例如問他為什麼天空是藍色的

這時的 LLM 就會開始思考，自己是教學型的老師，要引導學生思考，不能直接說答案，就會開始引導說光線的波長不一樣導致…

## Tool

LLM 大模型沒辦法感知現在外面的世界，例如問了「今天台北的天氣怎樣？」，他會回答「抱歉，無法得知現在的天氣訊息，我的知識庫截止到 2026年6月，無法提供當前的天氣資料」

為什麼？因為大模型只是一個文字接龍遊戲，他的能力根據訓練數據來預測下一個詞，他沒辦法真的去查天氣預報網站拿到即時的天氣資料，那這時候就是 Tool 的發揮機會了

Tool 本質上就像一個 function (函式)，一樣給他輸入，他就會給你輸出，例如用了一個天氣查詢工具，他的輸入可能會需要兩個 parameter (參數)，分別是城市和日期，他可能中間會去找一個氣象局的接口尋找資料，後續輸出就會是天氣和溫度

那依照剛剛的例子（天氣查詢）詳細解釋，我們可以把分出四個角色

- User 用戶
- LLM 大模型
- Tool 天氣查詢工具
- Platform 平台：這邊可以理解成一個傳話筒，因為用戶、大模型和工具，他們之間沒辦法直接對話，所以需要平台這個角色，來做傳遞訊息的工作，本質上他就是一段 code，用來負責上下傳達

Platform 在和 LLM 溝通時，除了把 User 問題給他，還會把 Tool list 給他，讓他知道現在有的工具列表，例如可能有天氣查詢、計算機等等，後續 LLM 會自動分析，結果他會發現他可以調用天氣查詢 Tool，注意 LLM 大模型沒辦法直接調用 Tool 工具，因為他的能力是輸出文本，所以需要透過 Platform 平台

## MCP

![image.png](https://github.com/chengyang1380/ProgrammingNotes/blob/main/Images/Programing/202606%20各種%20AI%20相關名詞/image%203.png?raw=true)

在上個例子中有兩個問題

1. Platform 要把 Tool list 傳給 LLM
2. 還要能 tool calling 

要做到這些，首先要把 tool 接入到 platform 裡，這樣 platform 就知道有哪些 tool list 以及每個參數及調用方法等等 

那有一個棘手的問題來了，這套接入的規範每個平台都不一樣，如果用的是 ChatGPT 就要按照 OpenAI 的接入規範，假如用的是 Calude，又需要按照 Anthropic 的接入規範，如果用的是 Gemini 又要按照 Google 的接入規範

問題就在於每個 platform 假如都要支援，就要一直寫新的規範，因此就有人想出要有一個統一的標準，讓所有的 platform 都是遵守這個標準，這樣 Tool 的開發者只需要一次 code，就可以支援所有 plaftorm，這就是 MCP，MCP 全名是 Model Context Protocol，用現實中的例子來比喻，就像現在的手機很多都用 type-c 接口，大家都用一樣的接口就會方便很多

## Agent

現在我們知道 LLM 可以透過 Tool 來感知外部世界，而 Tool 可以透過 MCP 來統一使用，照理說有了這兩個東西已經很強了，但實際上還有一個問題，我們讓 LLM 解決一個更有難度的問題，例如

Prompt 問題：今天我這邊的天氣如何？如果下雨的話，幫我查一下附近有沒有賣雨傘的店

Tool 工具：定位（可查詢用戶所在位置的經緯度）、天氣（透過經緯度查詢天氣）、店家（透過經緯度查詢附近的店家）

所以從 LLM 的角度來看，流程是這樣的：

1. LLM 思考，這問題需要先拿到用戶位置
2. 向 Platform 呼叫 Tool (定位)
3. 然後 Platform 拿到 Tool 結果（經緯度 -70, 40）
4. LLM 思考，拿到位置後下一步要呼叫該位置的天氣，然後會發現有一個天氣 Tool 可以用
5. 向 Platform 呼叫 Tool (天氣) 
6. 平台拿到結果有雨
7. LLM 在思考，發現下雨了，根據 User 需求，下雨的話要搜尋雨傘店家，發現有店家 Tool 可用
8. 向 Platform 呼叫 Tool (店家）
9. 平台拿到結果附近一百公尺有便利商店賣傘
10. LLM 在思考，目標已經全部達成
11. 可以生成最終答案了

![image.png](https://github.com/chengyang1380/ProgrammingNotes/blob/main/Images/Programing/202606%20各種%20AI%20相關名詞/image%204.png?raw=true)

整個流程已經不是單次簡單的調用 Tool 流程了，LLM 需要一直不斷思考當下情況，並決定下一步該做什麼，從某種程度來說 LLM 已經有一定的自主規劃能力，並自主調用相對應 Tool，這樣的系統為 Agent

現在市面上常見的就是 Codex, Claude Code, Gemini CLI

建構模式有 ReAct (Reason + Action) 和 Plan And Execute

**ReAct: loop:**

- **Thought (思考)**：現在的狀況是什麼？我下一步該做什麼？
- **Action (行動)**：呼叫某個 Tool 來執行動作。
- **Observation (觀察)**：檢視 Tool 回傳的結果。如果發現資料有缺或發生 Error，它不會放棄，而是回到 Thought 階段重新思考替代方案。

**Plan And Execute:**

1. Plan (整體規劃)：
由一個擁有強大推理能力的大模型擔任「規劃者 (Planner)」。它不急著動手，而是先將使用者的最終目標，拆解成一份有邏輯、有順序的「子任務清單」（就像是開出所有的 Jira Tickets）
2. Execute (逐一執行)：
將這些子任務依序發包給底層的「執行者 (Executor)」。這些執行者通常只需要專注在眼前的單一任務，甚至可以用較小、較便宜的模型來處理，執行完畢後只需回報結果，不用管整個大專案長怎樣。
3. Repalning (動態重擬 - 現代架構擴充)：
在每個子任務執行完畢後，系統會檢查結果。如果遇到預期外的錯誤（例如某個 API 掛了查不到資料），系統會把情況回報給規劃者，讓它即時修改接下來的計畫。

如果說 ReAct 是「走一步算一步，邊做邊想」，那麼 Plan and Execute 就是「先畫好系統架構圖，再按表操課發包施工」。

| **比較維度** | **ReAct (邊想邊做)** | **Plan and Execute (先想後做)** |
| --- | --- | --- |
| **開發情境比喻** | 像是在 Playground 或設中斷點 Debug，看著印出來的變數決定下一行要改什麼。 | 像是先開好 System Design 文件與所有 API Spec，再依序實作。 |
| **運作方式** | 思考 ➡️ 行動 ➡️ 觀察 (單一迴圈) | 擬定完整步驟 ➡️ 逐一獨立執行 |
| **強項** | 反應靈活，遇到突發錯誤可以馬上轉彎。 | 掌控全局，適合需要很多步驟的複雜長線任務。 |
| **致命傷** | 任務一長容易忘記初衷，耗費大量 Token。 | 如果第一步的計畫就規劃錯了，後面可能會全盤皆輸。 |

現在最先進的 Agent 框架（例如 LangGraph 或 AutoGen）通常不會只用單一模式，而是「用 Plan and Execute 控管大局，用 ReAct 執行單一任務」

## Agent Skill

我們現在知道 Agent 可以自主規劃並且調用工具，持續工作直到完成任務，但在實際高頻使用中會遇到一個痛點。

舉個例子，假設希望 LLM 成為你出門前的小助手，每次出門前都幫我們查一下天氣並提醒帶東西，你肯定有一個出門習慣，比如下雨會帶傘，太陽太大會戴帽子，空氣差會戴口罩，風大穿防風外套，無論如何要帶手機。

不僅如此可能還有強迫症，希望他的回答不要太囉唆，必須按照特定格式輸出，先來一句總結，後續再列出要帶的物品清單，並說明原因。

如果再沒有任何設定的情況下，假設問了一句「我要出門了，該帶些什麼？」，Angent 雖然會查天氣，但不知道剛剛上述的那些私人規則和格式要求，大概率會給一堆廢話，他無法依據剛剛的習慣來判斷要帶什麼，也沒有辦法按照想要的格式，畢竟都不知道

為了得到滿意的結果，不得每次要提問時，在 prompt 上加上一大串規則，試想每次出門都要求這些也太麻煩，這個情境就需要 Agent Skikk 登場了

Agent Skill 本質上就像提前給 Agent 一份說明文件，像剛剛的出門情境，就可以先把那些規範寫成 Agent Skill，他是一份 markdown，而他可以分成兩層

- 封面 - meta data，通常有 name 和 description，告訴 agent 這 skill 叫什麼以及大概怎麼用
- 指令層 - 主要向 agent 說明白要怎麼做，格式可以自行定義

完成這個 skill 後，要存在 local，拿 Claude Code 舉例，要放在 `.claude/skills` 資料夾裡，而裡面還有一些小規範，要先建立 skill 的資料夾，例如我們這個 skill name 叫做 go-out-checklist，這個資料夾名稱就叫做 go-out-checklist，然後其內容，就是我們的 skill，檔名記得叫 SKILL.md，這是 Agent Skill 的規範，要大寫，最後就會變成

```swift
skills/
├── go-out-checklist/
├──── SKILL.md
```

複雜一點的做法還可以在 `SKILL.md` 那層加上其他資料夾，例如 `scripts/` , `examples/`

之後在使用 ai 時，只要告訴他「我要出門了，要帶什麼？」他就會開始 tool calling 呼叫定位工具什麼的，一連串的完成事情，其中的事情都在 skill 裡定義的很清楚了。

所以 skill 本質上就是一種文件，給 agent 看的說明文件，當然還有其他玩法，run script, execute code, 引用資源等等，其他細節可參考

## Summary

LLM - 大模型，AI 技術的核心

Token - LLM 處理 data 的基本單位

Context - LLM 每次處理任務時接收到的訊息總和，可以想像是 LLM 臨時的記憶體，裝這歷史紀錄、系統規則以及當前輸入等等，這些數據的基本單位都是 Token

Context Window - LLM 的 Context 最多能儲存的 Token 量

Prompt - 用戶或系統當前給 LLM 下達的具體指令或問題，主要分為 User Prompt （代表用戶給 LLM 的輸入） 以及 System Prompt （類似開發者在後台配置給 LLM 人設和做事規則）

Tool - LLM 用來感知和影響外部環境的函數

MCP - 統一了 Tool 接入格式的標準協議

Agent - 能自主規劃和調用工具，直至解決用戶問題的程式

Agent Skill - 給 Agent 看的說明文件，主要規範做事步驟和規則
